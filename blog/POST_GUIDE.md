# Writing a blog post

Every post is one Markdown file in `blog/posts/`, named `YYYY-MM-DD-slug.md`.
Run `python3 blog/build.py` after adding or editing any post — it regenerates
`blog/<slug>/index.html`, `blog/index.html`, `sitemap.xml`, and `robots.txt`.

## Frontmatter (required on every post)

```yaml
---
title: "Full post title"
slug: your-post-slug
date: 2026-07-28
description: "SEO meta description, ~150-160 characters. Shows up in Google results."
deck: "One or two sentence subtitle shown under the headline on the post page."
tags: [Coordination, Bid Documents]
---
```

Optional:
- `og_image: "https://..."` — defaults to the Preempt Global logo if omitted.

`slug` must be unique across all posts and becomes the URL: `preemptglobal.com/blog/<slug>/`.
Use lowercase, hyphen-separated, no dates or special characters.

## Body: supported Markdown

- `## Heading` and `### Subheading` — section headings
- Plain paragraphs
- `**bold**`, `*italic*`, `` `code` ``
- `[link text](https://...)`
- `- item` or `1. item` — bulleted or numbered lists
- `> quoted line` — becomes a styled pull-quote (use for one strong, standalone
  sentence, not a real citation)
- `![alt text](image-url)` — a single image
- The **first paragraph** of the post should be prefixed with `!lede ` (e.g.
  `!lede Every large capital project already has...`) — this renders it larger
  and in muted color as the article's opening line. Use it once, at the top,
  and nowhere else.

Do not write raw `<p>`, `<h2>`, etc. by hand — write plain Markdown and the
build script converts it. Anything that starts with `<` is instead passed
through untouched (see the components below), so don't start an ordinary
paragraph with a `<`.

## Optional rich components

Paste these in verbatim, on their own block (blank line before and after,
no blank lines inside the block itself), anywhere in the body:

**Callout** (a boxed aside — use for one useful fact worth pulling out)
```html
<div class="callout">
  <div class="cl-label">Worth knowing</div>
  <p>The callout text goes here.</p>
</div>
```

**Mini stat report** (a small data table — use when you have concrete before/after numbers)
```html
<div class="mini-report">
  <div class="rh"><span class="title">LABEL</span><span class="meta">context</span></div>
  <div class="mini-row"><span class="k">Key</span><span class="v">Value</span></div>
  <div class="mini-row"><span class="k">Key</span><span class="v hot">Value</span></div>
</div>
```
(`class="v hot"` renders the value in the site's alert red — use it for the
number you most want to stand out, e.g. a dollar figure.)

**Takeaways box** (use once, near the end of most posts, 3-5 bullets)
```html
<div class="takeaways">
  <h3>Key takeaways</h3>
  <ul>
    <li>Point one.</li>
    <li>Point two.</li>
  </ul>
</div>
```

None of these are required — a post that's just headings, paragraphs, and
lists is completely fine.

## Feature image (optional, usually skip it)

Most posts should NOT have a header illustration — the template looks
complete without one. Only add one for a post where a specific diagram
genuinely adds something (like an actual conflict/exhibit visualization).

To add one, hand-write an SVG matching the site's navy/gold blueprint style
(see `blog/posts/2026-07-20-architect-qc-vs-owner-review.feature.html` for
a full example) and save it as `blog/posts/YYYY-MM-DD-slug.feature.html` —
same base filename as the post, `.feature.html` instead of `.md`. The build
script picks it up automatically if that file exists.

## What stays fixed on every post (don't try to change per-post)

The nav, the "Send a set for audit" CTA block, the Joe Rossi author box,
the share row, and the "related posts" grid (auto-computed from the 3 most
recent other posts) are all generated by `blog/template.html` — they don't
need to be, and can't be, set from the Markdown file.
